# UI Design Standard — Anti-AI-Slop Guideline

> **이 문서는 프론트엔드 코드 작성 시 반드시 적용할 자가 비판 루프 프로세스입니다.**
> UI/HTML/CSS/JS 코드를 작성할 때마다 아래 프로세스를 거쳐야 합니다.

---

## 역할 분리 (GAN 구조)

코드 작성 시 두 페르소나를 철저히 분리하여 사고한다.

- **Generator (생성자)**: 요구사항을 만족하는 첫 번째 구현을 완성한다.
- **Evaluator (평가자)**: 생성된 결과물을 냉혹하게 비판한다. 자화자찬 금지. 제3자의 시각으로 약점을 찾아낸다.

---

## 4대 평가 기준 (우선순위 순)

### 1. 독창성 (Originality) — 가중치: 최고

- 뻔한 템플릿 구조를 벗어났는가?
- 완전히 새로운 레이아웃 관점(비대칭, 타이포그래피 주도, 공간 활용 등)을 시도했는가?

### 🚨 절대 금지 (AI Slop 패널티)

아래 요소가 하나라도 있으면 **즉시 감점 → 디자인 전면 수정**:

- 보라색·파란색·청록색 그라데이션 배경
- 흰색 카드 + 둥근 모서리(border-radius ≥ 12px) 남발
- 기본 구글 폰트 조합(Roboto + 파란 버튼)
- 가운데 정렬된 히어로 섹션 + 큰 Heading + 서브텍스트 + CTA 버튼 구조
- 아이콘 + 텍스트 3열 카드 그리드 (기능 소개용)
- 별/하트 아이콘이 포함된 리뷰 섹션
- "Made with ❤️" 푸터

### 2. 디자인 퀄리티 (Design Quality) — 가중치: 최고

- 전체 디자인이 파편화되지 않고 하나의 일관된 비주얼 언어를 유지하는가?
- 색상 팔레트가 3가지 이하로 절제되어 있는가?
- 여백과 리듬이 의도적으로 설계되어 있는가?

### 3. 완성도 (Craft) — 가중치: 보통

- 타이포그래피: 폰트 크기 위계(hierarchy)가 명확한가?
- 색상 대비: WCAG AA 기준(4.5:1) 이상인가?
- 인터랙션: hover/focus/active 상태가 모두 정의되어 있는가?

### 4. 기능성 (Functionality) — 가중치: 보통

- 사용자가 헤매지 않고 직관적으로 사용할 수 있는가?
- 모바일/리사이즈 시에도 레이아웃이 깨지지 않는가?

## 작업 프로세스 (자가 개선 루프)

### Step 1 — 초안 생성

요구사항을 만족하는 첫 번째 구현을 작성한다.

### Step 2 — 평가자 시점에서 자가 비판

작성한 코드를 실제로 렌더링된 모습으로 상상하고, 4대 기준 기반으로 약점을 리스트업한다.

예시 비판 포인트:
- "이 카드 레이아웃은 전형적인 SaaS 랜딩 페이지처럼 보인다 → 구조 변경 필요"
- "파란 그라데이션 배경이 AI 생성 느낌을 준다 → 단색 또는 질감 기반으로 교체"
- "hover 효과가 없다 → 완성도 미달"

### Step 3 — 창의적 도약 및 수정

비판을 단순 개선으로 해소하지 말고, 뻔한 접근 방식을 버리고 구조 자체를 재검토한다.

- 카드 그리드 → 타이포그래피 중심 리스트로 재구성
- 중앙 정렬 히어로 → 비대칭 분할 레이아웃으로 재구성
- 색상 그라데이션 → 단색 + 질감(noise, grain) 또는 모노크롬 강조

### Step 4 — 컨텍스트 관리

코드가 길어지면 핵심 로직에만 집중하고, 반복 패턴은 CSS 변수/클래스로 추상화한다. 인라인 스타일 남발 금지.

## 최종 출력 형식

UI 코드를 제출할 때는 다음 형식으로 사고 과정을 먼저 요약한 뒤 코드를 제시한다:

```
[Evaluator 비판]
- (비판 포인트 1)
- (비판 포인트 2)

[창의적 도약]
- (어떤 방향으로 재설계했는지 1~2줄)

[최종 코드]
(코드)
```

## 이 프로젝트의 기존 디자인 시스템

기존 UI를 수정하거나 확장할 때는 아래 토큰을 따른다:

| 토큰 | 값 |
|------|----|
| 배경 | `#0a0a14` |
| 강조색 | `#8b5cf6` (바이올렛) |
| 보조 텍스트 | `#7a7a90` |
| 폰트 | 시스템 폰트 스택 (프레임워크 없음) |
| 언어 | 한국어 UI |

위 시스템과 충돌하는 새 디자인을 제안할 경우, 충돌 여부를 명시하고 사용자 확인을 받을 것.
